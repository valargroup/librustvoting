# Full Voting Round Lifecycle in Zashi

## Context

Zashi currently only handles two states: no active round (show "Create Test Round") and ACTIVE round (delegation → voting → shares). Rounds transition ACTIVE → TALLYING → FINALIZED automatically on-chain, but the app can't see those states — `/rounds/active` returns 404 once a round leaves ACTIVE. For TestFlight testing with a deployed chain, we need visibility into all phases and the ability to cycle through rounds without restarting services.

Additionally, all server URLs (chain API, helper server, IMT server) are hardcoded to localhost or a specific IP. These need to be configurable for remote deployment.

The `create-voting-session` endpoint has **zero authorization** — any client can create voting sessions. This needs to be gated on the SDK side before deploying.

### Ceremony prerequisite (new)

Since this plan was written, a **key exchange ceremony** was added to main. `CreateVotingSession` now requires `ceremony.Status == CONFIRMED` — without it, the chain rejects session creation with `ErrCeremonyNotReady`.

For a single-validator local setup, the ceremony completes automatically:

1. **Register Pallas key** — `POST /zally/v1/register-pallas-key` (validator's Pallas public key, generated by `init.sh` via `pallas-keygen`)
2. **Deal EA key** — `POST /zally/v1/deal-ea-key` (dealer ECIES-encrypts `ea_sk` to each validator's Pallas pk, publishes `ea_pk`)
3. **Auto-ack** — `PrepareProposal` automatically decrypts the ECIES payload and injects `MsgAckExecutiveAuthorityKey`. With one validator, a single block confirms the ceremony.

After step 3, the chain is in `CONFIRMED` state and round creation works. The EA public key (`ea_pk`) is pulled from ceremony state — Zashi never needs to know it.

**For local testing**, run `make ceremony` after the chain is up. This is a standalone Rust test that calls the existing `bootstrap_ceremony` function (idempotent — skips if already CONFIRMED):

```bash
# Terminal 1: init and start the chain
cd sdk && make init && make start

# Terminal 2: bootstrap ceremony (one-time, ~10s)
cd sdk && make ceremony

# Now the chain is ready for round creation from Zashi or test-e2e.
```

**Ceremony status** is queryable at `GET /zally/v1/ceremony`.

## Changes

### 0. Server URL configuration

Three URLs are currently hardcoded and need to be configurable for remote testing:

| URL                               | Current value               | Location                          |
| --------------------------------- | --------------------------- | --------------------------------- |
| Chain API (`baseURL`)             | `http://localhost:1318`     | `VotingAPIClientLiveKey.swift:11` |
| Helper server (`helperServerURL`) | `http://localhost:9091`     | `VotingAPIClientLiveKey.swift:14` |
| IMT server (`imtServerBaseUrl`)   | `http://46.101.255.48:3000` | `VotingStore.swift:22`            |

**Approach:** Store these in `Info.plist` build settings, readable at runtime via `Bundle.main`. This keeps the configuration static per build (TestFlight vs local dev) without requiring a UI settings screen for voting infra.

**`secant-mainnet-Info.plist`** — Add keys:

```xml
<key>ZallyChainURL</key>
<string>$(ZALLY_CHAIN_URL)</string>
<key>ZallyHelperServerURL</key>
<string>$(ZALLY_HELPER_SERVER_URL)</string>
<key>ZallyIMTServerURL</key>
<string>$(ZALLY_IMT_SERVER_URL)</string>
```

**Xcode build settings / `.xcconfig`** — Define defaults:

```
[ZALLY_CHAIN_URL =](http://localhost:1318)
[ZALLY_HELPER_SERVER_URL =](http://localhost:9091)
[ZALLY_IMT_SERVER_URL =](http://46.101.255.48:3000)
```

For TestFlight builds, override these in the CI/release xcconfig to point at the deployed servers.

**`VotingAPIClientLiveKey.swift`** — Replace hardcoded `ZallyAPIConfig` values:

```swift
public struct ZallyAPIConfig {
    public static var baseURL: String = {
        Bundle.main.infoDictionary?["ZallyChainURL"] as? String ?? "http://localhost:1318"
    }()
    public static var helperServerURL: String = {
        Bundle.main.infoDictionary?["ZallyHelperServerURL"] as? String ?? "http://localhost:9091"
    }()
}
```

**`VotingStore.swift`** — Replace hardcoded IMT URL:

```swift
private let imtServerBaseUrl: String = {
    Bundle.main.infoDictionary?["ZallyIMTServerURL"] as? String ?? "http://46.101.255.48:3000"
}()
```

### 1a. SDK-side gate: `enable_session_creation` config

The `create-voting-session` endpoint currently has no auth. Gate it behind a config flag in `app.toml`:

**`sdk/scripts/init.sh`** — Add to `[vote]` section:

```toml
# Allow the REST API to accept MsgCreateVotingSession requests.
# Disabled by default — enable only for test/admin deployments.
enable_session_creation = false
```

**`sdk/app/app.go`** — Read `vote.enable_session_creation` from `appOpts`, pass to `HandlerConfig`.

**`sdk/api/handler.go`** — Add `enableSessionCreation bool` to `HandlerConfig` and `Handler`. In `handleCreateVotingSession`, return 403 if the flag is false:

```go
if !h.enableSessionCreation {
    writeError(w, http.StatusForbidden, "session creation is disabled on this node")
    return
}
```

This follows the exact pattern of `ea_sk_path` — read from app.toml `[vote]` section, passed through to the API handler.

### 5. Rework initialization to handle all statuses

**`VotingStore.swift`** — The `.initialize` action changes:

1. Open voting DB (same as today)
2. Try `fetchActiveVotingSession()`
3. If 404 and `lastKnownRoundId` is non-empty → try `fetchRoundById(lastKnownRoundId)`
4. If session found, send `.activeSessionLoaded(session)`
5. `.activeSessionLoaded` routes based on `session.status`:
   - `.active` → proceed with notes/hotkey loading as today, start status polling
   - `.tallying` → set `screenStack = [.tallying]`, start status polling
   - `.finalized` → set `screenStack = [.results]`, fetch tally results
6. If no session at all → `activeSession` stays nil, show no-round UI

The key refactor: split the `.initialize` effect so notes/hotkey loading only happens for ACTIVE rounds. For TALLYING/FINALIZED, we skip straight to the appropriate screen.

### 6. Round status polling

**`VotingStore.swift`** — `.startRoundStatusPolling`:

- Poll `fetchRoundById(roundId)` every 5 seconds
- On status change → `.roundStatusUpdated(newStatus)`:
  - TALLYING → navigate to `.tallying` screen, keep polling
  - FINALIZED → navigate to `.results` screen, cancel polling, fetch tally results
- Cancelled on `.dismissFlow` and on FINALIZED

### 7. Snapshot height input

**`VotingStore.swift`** — `.createTestRound` action: replace hardcoded `3_235_470` with `UInt64(state.testRoundSnapshotHeight)`, error if invalid.

**`ProposalListView.swift`** — `createTestRoundCard()`: add a `TextField` for snapshot height above the create button, bound via `.setTestRoundSnapshotHeight` action.

### 8. Gate test controls behind feature flag

**`ProposalListView.swift`** — When `activeSession == nil`:

- Else → show a simple "No Active Voting Round" card without the create button

### 9. Screen routing

**`VotingView.swift`** — Add `.tallying` and `.results` cases to `screenView(for:)` and `screenId(_:)`.

### 10. Tallying view

**`ProposalListView.swift`** — Add a `TallyingView` struct (keeps voting views in one place):

### 11. Results view

**New file: `ResultsView.swift`** in the Voting feature directory.

Shows:

- Round header (title, "Finalized" badge)
- Per-proposal result cards: title, horizontal bar showing support vs oppose, ZEC amounts
  - Support amount = entries where `decision == 1`, Oppose = `decision == 2`
  - Bar proportional to support/oppose ratio
- If `featureFlags.votingTestControls`: "Start New Round" button at bottom

`.startNewRound` action: clears `activeSession`, `tallyResults`, `lastKnownRoundId`, and re-sends `.initialize`.
